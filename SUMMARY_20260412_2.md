# PSXRecomp v4 — Session Summary 2026-04-12 (Session 2)

## Phase: 4 (Boot to BIOS shell + memory card screen)

## What was done

### Interrupt delivery — three critical bugs fixed

1. **RFE after handler exit (IEc stuck at 0)**

   The BIOS exception handler's normal exit path (0xBFC10944) restores
   context from a kernel jmpbuf and does `jr $ra` — but does NOT
   execute RFE. On real hardware this is a longjmp to WaitEvent's
   caller, which later calls ExitCriticalSection. In our model,
   `return;` goes back to `psx_check_interrupts`, not to the jmpbuf
   target. IEc stayed 0 after the first interrupt, permanently
   disabling all further interrupts.

   **Fix:** After the handler returns, if IEc is still 0, apply RFE
   to pop the SR exception stack.

2. **GPR corruption from handler jmpbuf exit**

   The handler's exit path at 0xBFC10944 restores $ra, $sp, $s0-$s7
   from the kernel jmpbuf. On real hardware this is the non-local
   return to WaitEvent's caller. In our model, the interrupted code
   resumes with these wrong register values, causing unaligned access
   crashes.

   **Fix:** Save all 32 GPRs + HI/LO before dispatching the handler,
   restore them after. The interrupted code gets its exact
   pre-exception state back, matching real hardware (where the TCB
   save/restore mechanism preserves the full register context).

3. **VBlank stacking inside the handler**

   `psx_check_interrupts` is called at every block leader, including
   inside the exception handler. `dispatch_count` was incrementing
   during handler execution, firing VBlanks that accumulated in
   `i_stat`. After the handler returned with IEc=1, a new interrupt
   fired immediately, creating an infinite interrupt loop that starved
   the main code.

   **Fix:** Gate `dispatch_count++` and VBlank/timer ticks on
   `!in_exception`.

### DMA IRQ interrupt storm

   `update_master_irq()` in dma.c was **level-triggered**: it set
   `i_stat |= (1<<3)` whenever DICR master_flag was 1. Since the BIOS
   handler has no registered DMA callback, i_stat bit 3 was never
   acknowledged. Combined with the fixes above (IEc now re-enabled),
   this caused an interrupt on every single `psx_check_interrupts`
   call — complete interrupt storm, VSync wait never exits.

   **Fix:** Made `update_master_irq` edge-triggered (only set i_stat
   on 0→1 transition of master_flag). Additionally suppressed DMA
   completion IRQ posting entirely since transfers are synchronous in
   our model.

### Debug diagnostics added

   - `gpu_state` now reports: draw_area, draw_offset, GP0 command
     counters (nop/fill/draw/env/copy)
   - New `vram_peek` debug server command for inspecting arbitrary
     VRAM coordinates

## Current state

### What works

- **IEc properly toggles** — interrupts enable/disable correctly
  across exception entry/exit (SR=0x00000401 in main code, SR pushed
  on exception, RFE pops on exit)
- **VBlank interrupts fire** on schedule, handler processes EvCB
  chains, returns cleanly
- **Sony logo displays** and logo animation plays (VBlank-driven timer
  advances)
- **Shell main loop reached** — RA=0xBFC41D18 consistently, execution
  is in the VSync wait / draw loop
- **GP0 draw commands executing** — 217+ draw commands, 641+
  environment commands observed during shell operation
- **Zero dispatch misses** on clean run
- **Zero crashes** — runtime runs indefinitely

### What doesn't work yet

- **VRAM still shows logo fill (0x5AD6) in the display area.** The
  shell is sending draw commands (confirmed via GP0 counters) but
  the visible VRAM region does not change. The draws may be landing
  at coordinates outside the display area, or being clipped/masked.
  Draw area = [0,0,639,479], draw offset = [0,0].

- **Shell does not visually progress past the Sony logo.** The logo
  animation plays ("small little intro") and then the screen stays
  static. Internally the shell IS running (register sampling confirms
  the main loop), but the GPU rendering isn't producing visible
  output.

## Root cause analysis for remaining issue

The shell sends GP0 draw commands but VRAM doesn't change in the
display area (0,0)-(639,479). Possible causes (in order of
likelihood):

1. **Draws land in wrong framebuffer.** The shell uses double
   buffering. If GP1(0x05) display start isn't being called (or the
   display points at the logo's old framebuffer), draws to the
   back buffer are invisible. display_y=2 is the logo's setting;
   the shell should switch to display_y=0 or display_y=240.

2. **Draw commands are environment-only.** The GP0 stats show 217
   draws vs 641 env commands. The "draws" might be small primitives
   (the shell's floating particles) that are too sparse to notice in
   VRAM spot-checks. The main fill (GP0 0x02) is 0 — the shell
   might not use fill-rect for the background gradient.

3. **GP1 display swap function not executing correctly.** The shell
   calls 0x80059DA4 (normalizes to 0x1FC41DA4, IS in dispatch table)
   for VSync/display swap. This function has a timeout loop and calls
   another function. The swap might silently fail.

4. **VRAM writes silently dropped.** The mask bit or draw area
   settings might prevent writes. check_mask_bit or set_mask_bit
   in the GPU could be interfering. VRAM pixel 0x5AD6 has bit 15=0
   (mask clear), so this is unlikely.
